Hypothèses

Nous avons posé les hypothèses suivantes :

  • Il existe un seul type d'abonnement pour les membres : abonnement illimité.
  • La confirmation de participation aux séances doit se faire le jour même.
  • Le membre peut consulter uniquement les séances offertes le jour même.
  • Le membre doit payer le frais de service lors de l'inscription.
  • Le rapport de synthèse sera généré sous demande du gérant via le poste de travail de l'agent. L'agent ne va pas exécuter la création du rapport sans avoir la confirmation du gérant
  • Les professionels qui viennent offrir leurs services à #GYM sont déjà approuvés par le #GYM en avance. Il n'ont qu'à compléter l'étape d'inscription afin de devenir un professionnel prestataire de service(s).
  • Pour les membres, le frais d'adhésion est à payer chaque mois. Le Centre de Données n'enregistre aucune information sur la méthode de paiement
  • L'application permettra à un seul terminal d'effectuer les opérations. Il n'y a pas de connexions multiples de prévue initialement au serveur.
  • Les données sont stockées sur un poste différent que le terminal d'opérations. En d'autres mots, le Centre de Données #GYM est un serveur (connexion par réseau internet)
  • Les actions requises seront fait en saisissant les commandes correpondantes dans le terminal de l'ordinateur de l'agent
  • Le Centre de Données de #GYM a besoin de fonctionner 24h/7jrs
  • La console de l'agent qui va intéragir avec le Centre de Données sera développé par d'autres développeurs (en dehors de l'envergure - scope - du projet actuel). Les deux systèmes seront compatibles et respecteront les formats de commandes et de données.
  • Les numéros de compte membre ou profesionnel à 9 chiffres sont uniques, que la personne soit un professionel ou un membre.
  • Les membres ne sont jamais des professionnels et vice-versa.
  • La liste de validation sera effacée à chaque journée à minuit par un processus automatisé du Centre de données.
  • Le professionel accède automatiquement au centre, sans processus d'autorisation comme pour le membre.
  • Le paiement de frais de service peut être payé à partir de l'application mobile
  • Les séances des nouveaux services sont immédiatement instancié lors de la création d'un nouveau service, à condition que la date de début soit respectée pour la semaine en vigueur.
  • Les séances sont recyclées chaque semaine. C'est-à-dire que le numéro de séance reste valide de semaine en semaine. Les participants de la semaine échue sont retirés et facturés.
  • Les séances ne peuvent pas être mofidiées. On peut uniquement modifier les services qui génèrent les séances. Pour déplacer une séance par exemple, il faudra la suprimer et la créer. C'est ce que le processus de mise à jour effectuera.

Acteurs

Nous avons déterminé les acteurs suivants :

  • Agent : Un employé à la réception qui opère le terminal.
  • Client : Un client est un futur membre.
  • Membre : Un membre possède un abonnement actif.
  • Professionnel : Un prestataire de services professionnels ou un expert.
  • L'application Mobile #GYM: Une application mobile de #GYM qui offre l'opportunité d'effectuer les changements, l'inscription à une séance et autres tels traitements sans l'intervention d'un agent de réception.
  • Gérant:Un employé qui est le responsable de #GYM et qui supervise tous les autres employés et les activités quotidienne de #GYM
  • Tourniquet-Lecteur Code QR: Sert à la lecture du code QR d'un membre voulant accéder à #GYM.

Tâche 1 : Mise à jour des artéfacts

Glossaire

Agent
Une personne qui travaille à la réception du centre et qui aide les clients et les professionnels à effectuer diverses opérations (Ex : Inscription, annulation, changement de date, etc.).
Centre de Données
Le serveur qui stockera toutes les données sur les séances, les membres et les professionels. Le serveur central de #GYM
Code de service
Un code à sept chiffres qui indique le service à fournir par les professionnels aux membres dans #GYM.
Membre
Un client qui a adhéré à #GYM et qui possède aussi un numéro de membre.
Numéro de membre
Un identifiant unique assigné à chaque membre de #GYM lors de l’adhésion.
Numéro de professionnel
Un identifiant unique assigné à chaque professionnel qui veux fournir des services aux membres dans #GYM.
Professionnel
Une personne qui offre des services aux membres chez #GYM.
Services
Offerts par les professionnels aux membres. Les services comprennent mais ne se limitent pas à : entraineur personnel, yoga, conseils de nutrition, etc. Le terme activité pourra référer à un service.
Séance
Une période d’activité pendant laquelle un professionnel fournira un service aux membres qui se sont inscrit à cette séance.
Répertoire des Services
Une liste où se trouvent les services qui sont actuellement offerts dans #GYM par les professionnels aux membres
Inscription
Processus de réservation requis pour un membre qui désire participer à une séance de service (activité).

Risques

  1. Risques sévères
    • Bris de confidentialité des informations personnelles.
    • Perte de données sur le Centre de Données #GYM (pas de copie de sauvegarde).
    • Panne durant la période de lancement des procédures automatisées (ex: vendredi minuit).
  2. Risques importants
    • Ouverture d'une deuxième succursale. Hypothèse de travail : l'application permettra à un seul terminal d'effectuer les opérations.
    • Changement du besoin du client : permettre aux utiliseurs d'effectuer les opérations directement sur un portail web.
    • Perte du numéro de membre : aucune façon de retrouver le numéro d'utilisteur d'un compte membre ou profesionnel.
    • Fraude ou vol d'identité : absente de vérification à 2 facteurs (exemple nip)
  3. Risques éloignés
    • Des catégories de membres et d'abonnements pourraient être créés pour remplacer les accès illimités.
    • Ajout d'un deuxième terminal d'opération avec ou sans agent (borne interactive ou porte automatisée).
  4. Obstacles mineurs
    • Un professionnel est aussi un client du centre.
    • La machine responsable de l'exécution de l'ensemble du processus a échoué.
    • La firme de comptable RnB a du retard dans les enregistrements quotidiens.

Exigences non-fonctionnelles

  1. Le langage de programmation doit être Java.
  2. Le Centre de Donnée fonctionnera 24h/7 afin d'assurer l'accéssibilité des données à tout moment
  3. Les données à afficher apparaissent à l'écran de la console.
  4. Les données doivent être conservées dans plusieurs fichiers dans le Centre de données et non pas dans une base de données.
  5. Seulement les usagers autorisés peuvent accéder aux données stockés dans le Centre de Données de #GYM (Ex: Les agents)
  6. Le système sera fonctionnel avec un large nombre de membres et de services.
  7. Performance : temps de réponse rapide.
  8. Fiabilité : peu de pannes.

Analyse

Le Centre de données sera un serveur installé sur un porte de travail.
Ce poste aura une installation de Java, un disque dur (500Go) et une connexion internet pour établir des connexions avec le(s) poste(s) des agents.
Les informations seront stockées dans des fichiers de type data produits par sérialisation en JAVA (members.dat, professionals.dat, services.dat, registrations.dat, validations.dat, ...).
Le Centre de données acceptera les connexions entrantes des postes des agents.
Le système acceptera des requêtes sous format sérialisé par le biais des Sockets.
Les requêtes des membres ou des professionnels seront saisies par les agents dans une Console Java.
Les tâches suivantes seront informatisées : la vérification du numéro de membre, l'assignation d'un nouveau nombre du membre, calcul des inscriptions, génération de rapport synthèse, génération de fichier TEF.
En particulier, la procédure comptable principale (Génération de fichier TEF et le rapport synthèse) est fait en mode batch: La tâche sera lancée les vendredis à minuit.
Aussi, la mise à jour les enregistrements d'adhésion de chaque membre effectué par le système informatique RnB sera fait en mode batch: La tâche sera lancée chaque soir à 21h.
